証跡
自宅で 500 再現せず／サーバ側は Web UI で正常動作している前提があるなら経路差／/upload/_/ の resumable session URL が中継先で書き換わると multipart 整合が崩れ 500 が出やすい
検証

対策方向

次点 H2. 会社 Workspace 管理者ポリシー（NotebookLM upload 制限）

証跡

検証
HAR レスポンス body に policy / denied / forbidden 文字列、または Server: ヘッダが Google 以外
対策方向
Workspace 管理者へ NotebookLM API 利用許可申請。手動 UI も 500 なら確度上がる

あり得る H3. 会社 IP / アカウントの Google 側 rate-limit / abuse detection

証跡

検証

5. HAR 取得時の追加確認項目

HAR を送る前に目視で下記を確認して所見も添えると一発で確定する：

  • レスポンス header の Server: / Via: / X-Cache: — 中間装置の signature（Squid, BlueCoat/ProxySG, Zscaler, Netskope 等が出れば確定）
  • 500 レスポンス body が JSON か HTML か — HTML なら社内プロキシ／DLP のブロック画面が濃厚
  • upload URL の uploadType=resumable セッション URL が Google ドメイン（*.googleusercontent.com / *.googleapis.com）のまま完走しているか — 途中で別ホストに書き換えられていれば経路差し替え
  • 手動UIの upload (HAR-A) と API の upload (HAR-B) の URL / header が どこから違うか（行頭から diff を取る）

6. 切り分けマトリクス（結果別の次の一手）

観測有力仮説次の一手
A: 社内CA chain あり H1: SSL inspection が resumable upload を破壊 IT に inspection bypass を依頼 / CA bundle 注入で当座対処
B: 500 body が HTML / プロキシ signature H1 派生: DLP のコンテンツ検査ブロック 該当ホストの除外申請
手動UI も 500 H2: Workspace ポリシー or 上限 Workspace 管理者に NotebookLM 利用許可申請
手動UI は OK / API のみ 500 API パスのみ DLP / 自動化検知 HAR-A と HAR-B の header 差分を nlm_client.py に取り込み再試行
500 body に RATE_LIMIT / QUOTA H3: Google 側 abuse-detection 時間置き／別ネットワークで再現テスト
